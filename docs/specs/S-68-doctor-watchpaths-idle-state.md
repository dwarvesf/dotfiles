# S-68: Doctor accepts idle WatchPaths agents

Extends [S-66](S-66-dotfiles-sync-watcher-audit.md). The S-66 doctor's `agent-wp` check requires `state = running` on the WatchPaths agent (`com.truonghan.dotfiles-watcher`), which is only true during the brief seconds the agent is executing a tick. For a `WatchPaths`-only agent with no `KeepAlive` and no `StartInterval`, the steady state is `waiting` or `not running`, not `running`. The check fires false-positive `[err]` on healthy idle machines.

## Problem

S-66's doctor was written assuming both watchers would always show `state = running`. The fswatch sibling (`com.truonghan.dotfiles-watcher-fswatch`) does, because its plist has `KeepAlive=true` and a long-lived `fswatch -r` process keeps the agent alive.

The WatchPaths agent does NOT have `KeepAlive`. Its plist (regenerated by `run_onchange_after_dotfiles-watcher.sh.tmpl`) sets `WatchPaths` and `ThrottleInterval` only. launchd loads the agent at boot, then waits for an mtime change on one of the ~120 enumerated paths. Between events, `state = waiting` (or `not running` depending on macOS version). Only during the few seconds when a tick runs does `state = running`.

Doctor logic at `home/dot_local/bin/executable_dotfiles-watch-doctor` lines 51-57:

```sh
state=$(printf '%s\n' "$out" | sed -n 's/^[[:space:]]*state = //p' | head -1 | tr -d ' ')
if [ "$state" = "running" ]; then
    ok "agent: $label running"
else
    err "agent: $label loaded but state=${state:-unknown}" "dotfiles watch install"
fi
```

Repro: on this Mac mini today, immediately after `dotfiles watch install`, both agents are loaded. `dotfiles watch doctor` fires `[err] agent: com.truonghan.dotfiles-watcher loaded but state=notrunning` (with the inline `Fix: dotfiles watch install` suffix). The suggested "fix" is a no-op (the agent is already correctly loaded; re-installing changes nothing). The misdirection led the operator (this morning) to suspect the watcher was broken when it was actually idle waiting for the next file event.

Found while answering "did the watcher sync the new skills to the dotfile repo?" The doctor's false-positive was the first signal investigated, and was wrong. The real answer was a separate gap (new files were not in `chezmoi managed`, closed in S-67).

S-66 § Test row 2 says "the spec author claimed it would exit 0 on the Mini" but the matrix only tested `FAKE_LC_WP=running`, never `FAKE_LC_WP=loaded` (the idle case). The bug was undetected until live deploy.

## Solution

Treat `launchctl print` exit code as the source of truth for "agent is loaded":

- If `launchctl print "gui/$UID/$label"` exits 0, the agent is loaded; report `[ok]`.
- If it exits non-zero, the agent is not loaded; report `[err]` as today.

`state` is informational, not pathological. Include it in the `[ok]` line for context, but do not gate ok/err on the value.

Edge case: `launchctl print` succeeds but the state value is something unexpected (e.g., `state = errored`). Treat as `[warn]` with "unexpected state" message. Conservative middle-ground for hypothetical pathologies.

Concrete check:

```sh
check_agent() {
    label="$1"
    out=$(launchctl print "gui/$UID_/$label" 2>/dev/null) || {
        err "agent: $label not loaded" "dotfiles watch install"
        return
    }
    state=$(printf '%s\n' "$out" | sed -n 's/^[[:space:]]*state = //p' | head -1 | tr -d ' ')
    case "$state" in
        running|waiting|notrunning|idle|"")
            ok "agent: $label loaded (state=${state:-unknown})"
            ;;
        *)
            warn "agent: $label loaded but state=$state" "dotfiles watch install"
            ;;
    esac
}
```

Output format change: `[ok] agent: ... loaded (state=running)` vs the previous `[ok] agent: ... running`. Includes the state value in parentheses for visibility. Operator sees both "loaded" (the meaningful invariant) and "state=X" (the descriptive detail launchctl reports).

## Test

Three test impacts:

1. **Existing 4.1 (`test_doctor_clean`)** uses `FAKE_LC_WP=running` → state=running. Still passes (state in accept set, [ok]).
2. **Existing 4.2 (`test_doctor_agent_wp_missing`)** uses `FAKE_LC_WP=missing` → launchctl exits non-zero. Still passes (still [err]).
3. **New 4.9 (`test_doctor_agent_wp_idle`)** uses `FAKE_LC_WP=loaded` → state=waiting. Currently this would fail (doctor fires [err] on waiting). After the patch, this is the new normal case and should be [ok]:

```sh
test_doctor_agent_wp_idle() {
    local home out rc
    home=$(mktemp -d)
    _setup_doctor_home "$home"
    _seed_clean_fingerprint "$home"
    out=$(FAKE_MANAGED="" FAKE_LC_WP=loaded FAKE_LC_FS=running _run_doctor "$home" 2>&1)
    rc=$?
    if [ "$rc" -ne 0 ]; then
        echo "expected exit 0 for idle WP agent, got $rc"
        echo "$out"
        rm -rf "$home"
        return 1
    fi
    if echo "$out" | grep -qE '^\[(warn|err)\][[:space:]]+agent: com.truonghan.dotfiles-watcher\b'; then
        echo "idle WP agent should be [ok], not warn/err:"
        echo "$out"
        rm -rf "$home"
        return 1
    fi
    if ! echo "$out" | grep -qE '^\[ok\][[:space:]]+agent: com.truonghan.dotfiles-watcher loaded'; then
        echo "missing expected [ok] line for idle WP agent"
        echo "$out"
        rm -rf "$home"
        return 1
    fi
    rm -rf "$home"
    return 0
}
```

End-to-end on Mac mini: after `dotfiles watch install`, `dotfiles watch doctor` exits 0. Both agents emit `[ok] agent: ... loaded (state=...)`. The state value is whatever launchd reports (`waiting`, `running`, or `notrunning` are all acceptable). No `[err] agent-wp` false positive.

## Out of scope

- **Translating `state` values in the [ok] line.** Print the launchd-reported value as-is. Operators learn launchctl semantics by reading the value, and pasting the doctor output into a bug report stays useful.
- **Detecting crash-loop agents.** A `KeepAlive` agent that is crash-looping shows `state = running` plus a high `last exit code` and `relaunch count`. Out of scope for S-68; this is its own check (probe `last exit code != 0` AND `relaunch count > N`).
- **State transition between checks.** If the agent transitions `waiting → running → waiting` between launchctl print and the subsequent message, that is fine. The check is a snapshot, not a stream.
- **Updating S-66's spec or DoD.** S-66 stays as-is; S-68 is the follow-up. Same pattern S-65 was to S-64. The `extends` frontmatter is the cross-reference.

## Implementation notes

- **Why not normalize `notrunning` to `waiting`**: `state = not running` on macOS 26.x is just how launchd reports an idle WP agent. Normalizing in the doctor obscures the actual launchd output, which is useful when an operator pastes doctor output into a bug report.
- **Why not include `errored` or `crashed` in the accept set**: those are pathological. The `*)` branch catches them as `[warn]`. The fix command is unchanged (`dotfiles watch install`) because rebootstrap usually clears bad state.
- **Test mechanics**: `_make_fake_launchctl` (in `tests/dotfiles-watch.sh`) already emits `state = waiting` when `FAKE_LC_FS|FAKE_LC_WP=loaded`. No shim change needed; only a new test case wiring it up.
- **Why a separate SPEC and not an amend to S-66**: S-66 shipped 2026-05-13 morning, S-68 fixes a bug found 2026-05-13 evening. Same-day amend would be tidier in a vacuum, but the audit trail is more useful as a paired ship (S-66 + S-68) showing what the matrix missed. S-65 was already this pattern (post-ship doc sweep for S-64).
